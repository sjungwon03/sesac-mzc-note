# 인덱스와 뷰

## 1. 인덱스 (Index) 개요

### 1.1 인덱스란?

테이블의 특정 칼럼을 기준으로 데이터를 빠르게 찾기 위해 만드는 데이터베이스 객체이다.

책의 색인처럼 원하는 데이터를 찾기 위한 위치 정보를 따로 관리한다.

### 1.2 사용하는 이유

- `SELECT` 문의 검색 속도를 향상
- 조건 검색, 정렬, 조인 성능 개선에 도움
- 대량 데이터에서 Full Table Scan을 줄일 수 있음

### 1.3 장단점

| 구분 | 설명 |
| --- | --- |
| 장점 | 조회 성능 향상, 검색 비용 감소 |
| 단점 | 추가 저장 공간 필요, `INSERT`/`UPDATE`/`DELETE` 시 인덱스 갱신 비용 발생 |

**핵심:** 인덱스는 조회 성능을 높일 수 있지만, 데이터 변경이 많은 테이블에 무분별하게 만들면 오히려 성능이 떨어질 수 있다.

---

## 2. 클러스터 인덱스 (Clustered Index)

### 2.1 정의

실제 데이터가 인덱스 기준으로 정렬되어 저장되는 인덱스이다.

MySQL InnoDB에서는 기본키(`PRIMARY KEY`)가 클러스터 인덱스 역할을 한다.

### 2.2 특징

- 테이블당 하나만 존재
- 기본키를 기준으로 데이터가 정렬됨
- 범위 검색에 유리
- 중간 위치에 데이터가 삽입되면 페이지 분할 비용이 발생할 수 있음

### 2.3 예시

```sql
CREATE TABLE members (
    mem_id INT PRIMARY KEY,
    mem_name VARCHAR(50),
    addr VARCHAR(100)
);
```

`mem_id`는 기본키이므로 클러스터 인덱스가 생성된다.

```sql
SELECT *
FROM members
WHERE mem_id = 10;
```

기본키 조건으로 조회하므로 인덱스를 효율적으로 사용할 수 있다.

---

## 3. 세컨더리 인덱스 (Secondary Index)

### 3.1 정의

기본키 외의 칼럼에 추가로 생성하는 보조 인덱스이다.

실제 데이터의 물리적 순서를 바꾸지 않고, 별도의 인덱스 구조를 만든다.

### 3.2 일반 인덱스

중복 값을 허용하는 기본적인 보조 인덱스이다.

```sql
CREATE INDEX idx_members_name
ON members (mem_name);
```

또는 다음처럼 작성할 수 있다.

```sql
ALTER TABLE members
ADD INDEX idx_members_name (mem_name);
```

### 3.3 고유 인덱스

중복 값을 허용하지 않는 인덱스이다.

```sql
CREATE UNIQUE INDEX idx_members_email
ON members (email);
```

검색 성능 향상과 중복 방지 역할을 함께 수행한다.

---

## 4. 복합 인덱스

### 4.1 정의

두 개 이상의 칼럼을 묶어 생성하는 인덱스이다.

```sql
CREATE INDEX idx_members_addr_name
ON members (addr, mem_name);
```

### 4.2 칼럼 순서의 중요성

복합 인덱스는 왼쪽 칼럼부터 순서대로 사용된다.

```sql
-- 인덱스 사용 가능
WHERE addr = '서울'

-- 인덱스 사용 가능
WHERE addr = '서울' AND mem_name = '김철수'

-- 인덱스 사용 어려움
WHERE mem_name = '김철수'
```

**핵심:** 자주 조회하는 조건과 선별도가 높은 칼럼을 고려해 인덱스 순서를 정해야 한다.

---

## 5. 인덱스 사용 시 주의사항

### 5.1 인덱스 칼럼 가공 금지

인덱스가 걸린 칼럼을 함수나 연산으로 가공하면 인덱스를 사용하기 어렵다.

```sql
-- 인덱스 사용 어려움
WHERE mem_id * 2 >= 1000;

-- 인덱스 사용 가능성 높음
WHERE mem_id >= 1000 / 2;
```

### 5.2 너무 넓은 범위 조건 주의

조건에 해당하는 데이터가 너무 많으면 DBMS가 인덱스보다 전체 테이블 조회를 선택할 수 있다.

```sql
WHERE mem_id >= 1;
```

### 5.3 무분별한 인덱스 생성 금지

- 인덱스마다 저장 공간이 필요함
- 데이터 변경 시 모든 관련 인덱스를 갱신해야 함
- 사용하지 않는 인덱스는 성능 저하 원인이 될 수 있음

---

## 6. 인덱스 확인 및 삭제

### 6.1 인덱스 확인

```sql
SHOW INDEX FROM members;
```

### 6.2 인덱스 삭제

```sql
DROP INDEX idx_members_name ON members;
```

또는 다음처럼 작성할 수 있다.

```sql
ALTER TABLE members
DROP INDEX idx_members_name;
```

기본키 인덱스는 일반 인덱스처럼 삭제하지 않고 기본키 제약조건을 삭제해야 한다.

```sql
ALTER TABLE members
DROP PRIMARY KEY;
```

## 7. 뷰 (View) 개요

### 7.1 뷰란?

하나 이상의 테이블을 기반으로 만든 가상 테이블이다.

실제 데이터를 저장하지 않고, 조회 SQL 문을 저장해 두었다가 사용할 때 실행한다.

### 7.2 사용하는 이유

- 복잡한 쿼리를 단순하게 재사용
- 필요한 칼럼만 공개해 보안성 향상
- 사용자에게 일관된 조회 구조 제공

---

## 8. 뷰 생성과 사용

### 8.1 뷰 생성

```sql
CREATE VIEW v_members AS
SELECT mem_id, mem_name, addr
FROM members;
```

### 8.2 뷰 조회

```sql
SELECT *
FROM v_members;
```

일반 테이블처럼 `SELECT` 문에서 사용할 수 있다.

### 8.3 뷰 삭제

```sql
DROP VIEW v_members;
```

---

## 9. 뷰 사용 시 주의사항

- 뷰는 자체 데이터를 저장하지 않음
- 뷰에 별도의 인덱스를 만들 수 없음
- 복잡한 뷰는 오히려 성능 저하를 일으킬 수 있음
- 뷰를 통한 `INSERT`, `UPDATE`, `DELETE`는 제약이 많음

---

## 10. 핵심 정리

- 인덱스는 조회 성능을 높이기 위한 데이터베이스 객체이다.
- 기본키는 InnoDB에서 클러스터 인덱스로 동작한다.
- 세컨더리 인덱스는 필요한 칼럼에 추가로 생성하는 보조 인덱스이다.
- 복합 인덱스는 칼럼 순서가 중요하다.
- 뷰는 SQL 조회 결과를 가상 테이블처럼 사용하는 객체이다.
