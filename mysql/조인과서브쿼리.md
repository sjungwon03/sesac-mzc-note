# 조인과 서브쿼리 학습 노트

## 1. 조인 (Join) 개요

### 1.1 정의

2개 이상의 테이블을 서로 묶어서 동시에 데이터를 조회하는 방식

### 1.2 필요성

데이터베이스 설계 시 데이터 중복을 막기 위해 테이블을 분리(정규화)하므로, 필요한 정보가 여러 테이블에 흩어져 있을 때 이를 결합하기 위해 사용

### 1.3 조인 종류

1. 크로스 조인 (Cross Join)
2. 내부 조인 (Inner Join)
3. 외부 조인 (Outer Join)

### 1.4 주의사항

**1) 모호성(Ambiguous) 에러 방지**
- 두 테이블에 동일한 이름의 칼럼이 존재하면, `SELECT` 절이나 `ON` 절에 테이블명 명시 필수
- 에러 예시: `Error Code: 1052. Column 'dept_id' in field list is ambiguous`

**2) 관계성**
- 크로스 조인을 제외한 대부분의 조인은 1:N 관계로 연결된 테이블을 대상으로 수행

### 1.5 기본 문법

```sql
SELECT 조회_칼럼
FROM 첫_번째_테이블
JOIN 두_번째_테이블
ON 조인_조건
WHERE 조회_조건;
```

---

## 2. 크로스 조인 (Cross Join)

### 2.1 정의

"상호 조인" 또는 "Cartesian Product (카테시안 곱)"

### 2.2 특징

- 첫 번째 테이블의 모든 행을 두 번째 테이블의 모든 행과 무조건 조인
- 1:N 관계가 필요 없음
- A 테이블 10행 × B 테이블 5행 = 50행 결과
- 실무에서 거의 사용하지 않음 (테스트 데이터 생성용)
- `ON` 절 생략 또는 잘못 작성 시 CROSS JOIN으로 처리됨 (주의!)

### 2.3 문법

```sql
SELECT 조회_칼럼
FROM 첫_번째_테이블
CROSS JOIN 두_번째_테이블;
```

### 2.4 예시

```sql
SELECT e.emp_id, e.emp_name, d.dept_name
FROM departments d
CROSS JOIN employees e;
```

**결과:**
- departments: 2행 (영업부, 인사부)
- employees: 4행
- 결과: 8행 (모든 조합)

---

## 3. 내부 조인 (Inner Join)

### 3.1 정의

두 테이블 간에 조인 조건으로 지정한 칼럼의 값이 **공통적으로 일치하는 데이터만** 결합하여 추출

### 3.2 특징

- 어느 한 쪽에만 존재하는 값(매칭되지 않는 행)은 결과에서 **완전히 제외**
- 가장 일반적으로 사용하는 조인

### 3.3 문법

**두 개 테이블 조인**
```sql
SELECT 조회_칼럼
FROM 테이블1 INNER JOIN 테이블2
ON 조인조건;
```

**세 개 테이블 조인**
```sql
SELECT 조회_칼럼
FROM 테이블1 INNER JOIN 테이블2
ON 조인조건 INNER JOIN 테이블3
ON 조인조건;
```

### 3.4 예시

```sql
SELECT e.emp_id, e.emp_name, d.dept_name
FROM departments d
INNER JOIN employees e
ON d.dept_id = e.dept_id;
```

**결과:**
- "총무부", "기획부"는 employees 테이블에 없으므로 제외
- 매칭되는 데이터만 조회

---

## 4. 외부 조인 (Outer Join)

### 4.1 정의

한 테이블의 내용은 **모두 포함**하고, 다른 한 테이블의 내용은 **일치하는 정보만 포함**하는 조인 방식

### 4.2 특징

- 어느 한 쪽에만 존재하는 값도 조회 가능 (내부 조인에서는 불가능)
- 매칭되는 반대쪽 데이터가 없으면 `NULL` 값을 채움

### 4.3 종류

**1) 왼쪽 외부 조인 (LEFT OUTER JOIN)**
- 왼쪽 테이블의 데이터를 무조건 조회 결과에 포함

```sql
SELECT 조회_칼럼
FROM 테이블1 LEFT OUTER JOIN 테이블2
ON 조인조건;
```

**2) 오른쪽 외부 조인 (RIGHT OUTER JOIN)**
- 오른쪽 테이블의 데이터를 무조건 조회 결과에 포함

```sql
SELECT 조회_칼럼
FROM 테이블1 RIGHT OUTER JOIN 테이블2
ON 조인조건;
```

### 4.4 예시

```sql
SELECT e.emp_id, e.emp_name, d.dept_name
FROM departments d
LEFT OUTER JOIN employees e
ON d.dept_id = e.dept_id;
```

**결과:**
- departments 테이블의 모든 데이터 포함
- "총무부", "기획부" → emp_id, emp_name은 NULL로 표시

---

## 5. 조인 종류 비교

| 조인 종류 | 특징 | 매칭 실패 시 | 사용 빈도 |
|-----------|------|--------------|-----------|
| Cross Join | 모든 조합 | - | 거의 없음 |
| Inner Join | 공통 데이터만 | 제외 | 높음 |
| Left Outer Join | 왼쪽 전체 + 매칭 | NULL | 높음 |
| Right Outer Join | 오른쪽 전체 + 매칭 | NULL | 낮음 |

## 6. 서브쿼리 (Subquery) 개요

### 6.1 정의

하나의 메인 SQL문 안에 괄호 `()`로 삽입되어 결과를 제공하는 하위 쿼리

### 6.2 동작 원리

괄호 `()` 안의 서브쿼리가 먼저 실행 → 결과 반환 → 메인 쿼리 실행

### 6.3 필요한 상황 예시

**조회 대상:**
- 제품번호가 "1001"인 제품이 생산되는 공장에서 생산되는 제품 목록 조회

**분석:**
- 서브쿼리: 제품번호 "1001"의 공장 조회
- 메인쿼리: 해당 공장의 제품 목록 조회

### 6.4 구분 (3가지)

1. 중첩 서브쿼리 (Nested Subquery) - `WHERE` 절
2. 스칼라 서브쿼리 (Scalar Subquery) - `SELECT` 절
3. 인라인 뷰 (Inline View) - `FROM` 절

---

## 7. 중첩 서브쿼리 (Nested Subquery)

### 7.1 위치

`WHERE` 절에 위치 (가장 자주 사용)

### 7.2 종류

**1) 단일 행 서브쿼리 (Single Row Subquery)**

서브쿼리 결과가 **오직 1개의 행(1개의 값)**만 반환

**사용 경우:**
- 기본키(PK) 조건 검색
- 유니크(Unique) 칼럼 검색
- 통계 함수 (`AVG()`, `MAX()` 등) 결과

**연산자:**
- 일반 비교 연산자: `=`, `!=`, `>`, `>=`, `<`, `<=`

**예시:**
```sql
-- 평균 급여 이상을 받는 사원 조회
SELECT * FROM employees
WHERE salary > (SELECT AVG(salary) FROM employees);
```

**2) 다중 행 서브쿼리 (Multi Row Subquery)**

서브쿼리 결과가 **2개 이상의 행(여러 개의 값)** 반환

**연산자:**
| 연산자 | 설명 |
|--------|------|
| `IN` | 결과 중 하나라도 일치하면 참 (가장 많이 사용) |
| `ANY` | 결과 중 하나라도 만족하면 참 |
| `ALL` | 결과 전부와 일치하면 참 |

**주의:** 단일 행 연산자 사용 시 에러 발생

**예시:**
```sql
-- 부서번호 1인 부서 사원들과 같은 직급을 가진 사원 조회
SELECT * FROM employees
WHERE position IN (SELECT position FROM employees WHERE dept_id = 1);
```

## 8. 스칼라 서브쿼리 (Scalar Subquery)

### 8.1 위치

`SELECT` 절에 위치

### 8.2 특징

- `SELECT` 절에 마치 하나의 칼럼처럼 위치
- **반드시 하나의 값(단일 행, 단일 열)**만 반환해야 함
- 일치하는 데이터가 없으면 `NULL` 반환

### 8.3 예시

```sql
-- 상호연관 서브쿼리 (메인쿼리의 e.dept_id 값을 서브쿼리가 전달받아 비교)
SELECT emp_name,
       (SELECT dept_name FROM departments WHERE dept_id = e.dept_id) AS dept_name
FROM employees e
WHERE emp_id = 1001;
```

### 8.4 성능 주의사항

**RBAR (Row By Agonizing Row) 문제:**
- 메인쿼리의 행 데이터가 많은 경우, 매 행마다 서브쿼리가 재실행
- 성능 저하 발생

**해결 방법:**
- 조인(JOIN)으로 변경
- **핵심:** 스칼라 서브쿼리는 매칭 실패 시 `NULL` 반환 → `LEFT OUTER JOIN`으로 변경해야 데이터 유실 없음

```sql
-- 스칼라 서브쿼리를 LEFT OUTER JOIN으로 변환
SELECT e.emp_name, d.dept_name
FROM employees e LEFT OUTER JOIN departments d
ON e.dept_id = d.dept_id;
```

## 9. 인라인 뷰 (Inline View)

### 9.1 위치

`FROM` 절에 위치

### 9.2 특징

- `FROM` 절에서 마치 하나의 테이블처럼 취급 (가상 테이블)
- **반드시 별명(Alias)을 지정**해야 작동
- 인라인 뷰 내부에서 조회한 칼럼만 메인쿼리에서 사용 가능

### 9.3 MySQL 활용 상황

**1) 연산 순서 제어**
- 대용량 테이블 전체 조인 시 성능 저하
- 인라인 뷰 내에서 `WHERE` 절로 필터링 후 조인

**2) 집계 데이터와의 조인**
- `GROUP BY`를 이용한 인라인 뷰(집계 데이터) 구한 후 원본 테이블과 조인

**예시:**
```sql
-- 각 부서의 평균 급여와 사원 정보 조인
SELECT e.emp_name, e.salary, d.avg_salary
FROM employees e
INNER JOIN (
    SELECT dept_id, AVG(salary) AS avg_salary
    FROM employees
    GROUP BY dept_id
) d ON e.dept_id = d.dept_id;
```

### 9.4 Oracle vs MySQL

| DBMS | 페이징 처리 |
|------|------------|
| MySQL | `LIMIT 3, 3` (간단) |
| Oracle | 인라인 뷰 + `rownum` (복잡) |

## 10. DML과 서브쿼리 확장 문법

### 10.1 INSERT 문 결합

대량 데이터 복사 - `VALUES` 절 대신 서브쿼리 사용

```sql
INSERT INTO backup_employees (emp_id, emp_name)
(SELECT emp_id, emp_name FROM employees WHERE hire_date < '2000-01-01');
```

### 10.2 UPDATE 문 결합

수정할 값이나 조건절에 서브쿼리 배치

```sql
UPDATE employees
SET salary = (SELECT AVG(salary) FROM employees)
WHERE emp_id = 1002;
```

### 10.3 DELETE 문 결합

삭제 조건을 서브쿼리로 설정

```sql
DELETE FROM employees
WHERE dept_id = (SELECT dept_id FROM departments WHERE dept_name = '기획부');
```

### 10.4 수정/삭제 서브쿼리의 한계

**문제점:**
- MySQL에서 동일한 테이블을 `UPDATE`/`DELETE` 하면서 조건절 서브쿼리에서 조회 시 에러 발생
- `Error Code: 1093. You can't specify target table '테이블명' for update in FROM clause`

**원인:**
- 데이터를 수정/삭제하면서 동시에 조회 → 락(Lock) 충돌 및 데이터 정합성 파괴

**우회 방법:**
- 서브쿼리를 인라인 뷰로 한 번 더 감싸서 임시 테이블로 처리

```sql
-- 에러 발생 쿼리
DELETE FROM employees
WHERE salary < (SELECT AVG(salary) FROM employees);

-- 우회 쿼리 (인라인 뷰로 감싸기)
DELETE FROM employees
WHERE salary < (
    SELECT avg_salary FROM (
        SELECT AVG(salary) AS avg_salary FROM employees
    ) AS temp
);
```

## 11. 실전 성능 팁

### 11.1 JOIN 쿼리 성능

**A JOIN B인 경우:**
- A: 소량 데이터, B: 대량 데이터
- A에 PK, B에 FK 설정
- JOIN 시 PK를 기준으로 FK를 찾아 빠름

**테이블 분류:**
- **Drive Table (드라이브 테이블):** A (소량)
- **Driven Table (드리븐 테이블):** B (대량)

### 11.2 성능 최적화 가이드

1. **스칼라 서브쿼리 → JOIN으로 변경** (대량 데이터 시)
2. **인라인 뷰로 필터링 먼저 수행** (대용량 조인 시)
3. **인덱스 활용** (PK, FK)
4. **적절한 조인 종류 선택** (Inner vs Outer)

---

## 12. 핵심 요약

### 조인 (Join)
- **Cross Join:** 모든 조합 (Cartesian Product)
- **Inner Join:** 공통 데이터만 (매칭 실패 시 제외)
- **Outer Join:** 한쪽 전체 + 매칭 (매칭 실패 시 NULL)

### 서브쿼리 (Subquery)
- **중첩 서브쿼리 (WHERE):** 단일 행 vs 다중 행 연산자 구분
- **스칼라 서브쿼리 (SELECT):** 단일 값 반환, 성능 주의 (RBAR)
- **인라인 뷰 (FROM):** 가상 테이블, 별명 필수

### 성능 팁
- 스칼라 서브쿼리 → LEFT OUTER JOIN으로 변경
- 대량 데이터 → 인라인 뷰로 필터링 후 조인
- Drive Table (소량) → Driven Table (대량) 순서
