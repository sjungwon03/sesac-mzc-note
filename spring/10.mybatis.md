# 10. MyBatis와 트랜잭션

## JDBC의 문제점

자바에서 DB를 다루려면 JDBC를 쓴다. 근데 JDBC는 반복 코드가 너무 많다. Connection 연결, PreparedStatement 생성, ResultSet 처리, 자원 반납, 예외 처리를 매번 써야 한다. 게다가 SQL문이 자바 코드 안에 문자열로 박혀 있어서 오타를 찾기도 어렵고, SQL 수정할 때마다 자바를 다시 컴파일해야 한다.

이 문제를 해결하려고 자바 코드와 SQL을 분리하고, 서로 매핑해주는 SQL Mapper Framework인 **MyBatis**가 탄생했다.

---

## MyBatis란?

SQL Mapping Framework다. 자바 코드와 SQL을 분리해서 처리해주는 영속성 프레임워크다. JDBC의 반복 코드를 내장 핸들러가 다 처리해주니까, 개발자는 XML 파일이나 어노테이션에 SQL만 작성하면 된다.

2010년에 iBatis 팀이 Google로 이동하면서 이름이 MyBatis로 바뀌었다. iBatis의 한계였던 동적 쿼리와 어노테이션 처리를 개선했다.

핵심 원리는 SQL을 XML에 별도로 작성하고, 식별자(id)를 붙여 메서드처럼 호출하는 방식이다.

## Mapper XML 기본 구조

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE mapper PUBLIC "-//mybatis.org//DTD Mapper 3.0//EN"
    "http://mybatis.org/dtd/mybatis-3-mapper.dtd">
<mapper namespace="Mapper_인터페이스_전체경로">

</mapper>
```

## 바인딩: `#{ }` vs `${ }`

자바에서 전달한 값을 쿼리에 적용하는 방식이다.

| 구분 | `#{ }` | `${ }` |
| ---- | ------ | ------ |
| 방식 | PreparedStatement | Statement |
| 동작 | `?` Place Holder에 값 바인딩 | 값을 그대로 문자열 치환 |
| SQL Injection | 안전 | 취약 |
| 문자열 처리 | 따옴표 자동 추가 | 따옴표 직접 처리 필요 |

원칙적으로 `#{ }`를 쓴다. `${ }`는 칼럼명이나 테이블명, 정렬 방향(`DESC`) 등을 전달할 때만 쓴다.

```xml
<!-- #{} 사용 -->
<select id="findUser" resultType="com.example.project.dto.UserDto">
    SELECT id, username FROM members WHERE username = #{username}
</select>

<!-- ${} 사용: 칼럼명, 정렬 방향 전달 -->
<select id="findAllUsers" resultType="com.example.project.dto.UserDto">
    SELECT id, username FROM members ORDER BY ${columnName} ${sortOrder}
</select>
```

## 데이터 조회: `<select>`

- `id`: Mapper 인터페이스의 메서드명과 일치
- `resultType`: 결과를 저장할 자바 클래스 전체 경로. 결과 행 개수와 상관없이 항상 단수형 객체 타입을 쓴다

```xml
<select id="findAllProducts" resultType="com.example.project.dto.ProductDto">
    SELECT id, name, price, stock FROM products
</select>
```

---

## 데이터 조작: `<insert>`, `<update>`, `<delete>`

- `id`는 Mapper 인터페이스 메서드명과 일치
- `resultType`은 작성하지 않는다
- `parameterType`은 생략 가능하다

```xml
<insert id="saveProduct" parameterType="com.example.project.dto.ProductDto">
    INSERT INTO products (name, price, stock) VALUES (#{name}, #{price}, #{stock})
</insert>

<update id="updateProductStock">
    UPDATE products SET stock = #{stock} WHERE id = #{id}
</update>

<delete id="deleteProduct">
    DELETE FROM products WHERE id = #{id}
</delete>
```

## `<resultMap>`

DB 칼럼명과 DTO 필드명이 다르거나, DTO 안에 또 다른 객체(1:1, 1:N 관계)가 있을 때 수동으로 매핑해주는 태그다. 조인 쿼리 결과를 객체 지향 구조로 바인딩할 때 주로 쓴다.

```java
@Getter
public class CommentDto {
    private Long commentId;
    private String content;
    private String writer;
}
```

```java
@Getter
public class BoardDetailDto {
    private Long boardId;
    private String title;
    private String content;
    private String author;
    private List<CommentDto> comments;
}
```

```xml
<resultMap id="BoardDetailMap" type="com.example.project.dto.BoardDetailDto">
    <id property="boardId" column="board_id" />
    <result property="title" column="title" />
    <result property="content" column="board_content" />
    <result property="author" column="author" />
    <collection property="comments" ofType="com.example.project.dto.CommentDto">
        <id property="commentId" column="comment_id" />
        <result property="content" column="comment_content" />
        <result property="writer" column="writer" />
    </collection>
</resultMap>

<select id="findBoardWithComments" resultMap="BoardDetailMap">
    SELECT B.id AS board_id, B.title, B.content AS board_content, B.author,
           C.id AS comment_id, C.content AS comment_content, C.writer
    FROM board B LEFT JOIN comments C ON B.id = C.board_id
    WHERE B.id = #{boardId}
</select>
```

## `useGeneratedKeys`

INSERT 직후에 AUTO_INCREMENT로 생성된 PK값을 바로 알아내는 방법이다. `keyProperty`에 지정한 DTO 필드에 값이 주입된다.

```xml
<insert id="saveAndReturnKey"
        useGeneratedKeys="true"
        keyProperty="id"
        parameterType="com.example.project.dto.ProductDto">
    INSERT INTO product (name, price, stock) VALUES (#{name}, #{price}, #{stock})
</insert>
```

```java
ProductDto target = new ProductDto(null, "최신형 모니터", 350000, 10);
productMapper.saveAndReturnKey(target);
Long generatedId = target.getId();  // DB가 생성한 PK값이 세팅됨
```

## `<sql>`과 `<include>`

여러 쿼리에서 공통으로 쓰는 SQL 조각을 `<sql>`에 저장하고, `<include>`로 가져다 쓴다.

```xml
<sql id="productColumns">id, name, price, stock, created_at, updated_at</sql>

<select id="findAll" resultType="com.example.project.dto.ProductDto">
    SELECT <include refid="productColumns" /> FROM products
</select>

<select id="findById" resultType="com.example.project.dto.ProductDto">
    SELECT <include refid="productColumns" /> FROM products WHERE id = #{id}
</select>
```

## 동적 쿼리

### `<if>`

조건식이 참일 때만 SQL 구문을 추가한다.

```xml
<select id="searchProducts" resultType="com.example.project.dto.ProductDto">
    SELECT id, name, price FROM products
    <where>
        <if test="name != null and name != ''">AND name LIKE CONCAT('%', #{name}, '%')</if>
        <if test="minPrice != null">AND price &gt;= #{minPrice}</if>
    </where>
</select>
```

XML 안에서 `>`, `<`는 태그로 오해되니까 `&gt;`, `&lt;`로 쓰거나 `<![CDATA[ ]]>`로 감싼다.

### `<where>`와 `<set>`

동적 쿼리를 조립하다 보면 `WHERE AND name = ...`이나 `SET price = 10, WHERE ...` 같은 문법 에러가 생길 수 있다. `<where>`는 앞의 `AND`/`OR`을 자동으로 제거하고, `<set>`은 마지막 콤마를 자동으로 제거한다.

```xml
<select id="searchSmart" resultType="com.example.project.dto.ProductDto">
    SELECT * FROM products
    <where>
        <if test="name != null">name = #{name}</if>
        <if test="address != null">AND address = #{address}</if>
    </where>
</select>

<update id="updateSmart">
    UPDATE products
    <set>
        <if test="name != null">name = #{name},</if>
        <if test="price != null">price = #{price},</if>
    </set>
    WHERE id = #{id}
</update>
```

### `<choose>`, `<when>`, `<otherwise>`

자바의 switch-case와 비슷하다. 여러 조건 중 하나만 선택한다.

```xml
<select id="findProductsWithSort" resultType="com.example.project.dto.ProductDto">
    SELECT id, name, price FROM products ORDER BY
    <choose>
        <when test="sortBy == 'price'">price DESC</when>
        <when test="sortBy == 'name'">name ASC</when>
        <otherwise>id DESC</otherwise>
    </choose>
</select>
```

### `<foreach>`

List나 Array를 반복하면서 SQL을 조립한다. `IN` 절이나 배치 INSERT에 쓴다.

| 속성 | 설명 |
| ---- | ---- |
| `item` | 반복 객체의 각 요소 |
| `collection` | 반복할 Collection 객체 |
| `open` | 시작할 때 붙일 문자 |
| `separator` | 요소 사이 구분자 |
| `close` | 끝날 때 붙일 문자 |

```xml
<select id="findProducts" resultType="com.example.project.dto.ProductDto">
    SELECT id, name, price, stock FROM products WHERE id IN
    <foreach collection="idList" item="id" open="(" separator="," close=")">
        #{id}
    </foreach>
</select>

<insert id="saveAllProducts">
    INSERT INTO products (name, price, stock) VALUES
    <foreach collection="productList" item="p" separator=",">
        (#{p.name}, #{p.price}, #{p.stock})
    </foreach>
</insert>
```

## 스프링 MyBatis 실습

**의존성:**
```groovy
runtimeOnly 'com.mysql:mysql-connector-j'
implementation 'org.mybatis.spring.boot:mybatis-spring-boot-starter:3.0.4'
```

**설정 (`application.yml`):**
```yaml
spring:
  datasource:
    driver-class-name: com.mysql.cj.jdbc.Driver
    url: jdbc:mysql://localhost:3306/edu_db?serverTimezone=Asia/Seoul&characterEncoding=UTF-8
    username: root
    password: 1234

mybatis:
  mapper-locations: classpath:mappers/**/*.xml
  configuration:
    map-underscore-to-camel-case: true
```

`map-underscore-to-camel-case: true`를 켜면 DB의 `created_at`을 Java의 `createdAt`에 자동 매핑해준다.

**DTO:**
```java
@AllArgsConstructor
@Getter
@Setter
public class ProductDto {
    private Long id;
    private String name;
    private int price;
    private int stock;
}
```

**Mapper 인터페이스:**
```java
@Mapper
public interface ProductMapper {
    List<ProductDto> findAll();
    void save(ProductDto productDto);
}
```

**Mapper XML:**
```xml
<mapper namespace="com.example.project.mapper.ProductMapper">
    <select id="findAll" resultType="com.example.project.dto.ProductDto">
        SELECT id, name, price, stock FROM products ORDER BY id DESC
    </select>
    <insert id="save" parameterType="com.example.project.dto.ProductDto">
        INSERT INTO products (name, price, stock) VALUES (#{name}, #{price}, #{stock})
    </insert>
</mapper>
```

**Service:**
```java
@Service
public class ProductService {

    private final ProductMapper productMapper;

    public ProductService(ProductMapper productMapper) {
        this.productMapper = productMapper;
    }

    public List<ProductDto> getAllProducts() {
        return productMapper.findAll();
    }

    public void registerProduct(ProductDto productDto) {
        productMapper.save(productDto);
    }
}
```

## 트랜잭션

트랜잭션은 여러 SQL 실행 단위를 하나로 묶는 논리적 단위다. 전부 성공하거나 전부 실패해야 한다(All or Nothing).

스프링은 두 가지 방식을 제공한다:

| 방식 | 설명 | 장점 | 단점 |
| ---- | ---- | ---- | ---- |
| 프로그램 방식 | 개발자가 직접 begin/commit/rollback | 범위 미세 조정 가능 | 비즈니스 로직과 섞임 |
| 선언적 방식 | `@Transactional` 어노테이션으로 위임 | 코드 깔끔, 간편 | 메서드 단위로만 조정 |

실무에서는 거의 선언적 방식을 쓴다.

---

## `@Transactional`

Service 계층에서 트랜잭션을 관리한다. 메서드나 클래스 위에 붙이면 스프링 AOP가 트랜잭션을 자동으로 처리한다.

클래스 레벨과 메서드 레벨에 동시에 있으면 메서드 레벨이 우선된다(Override).

### 주요 옵션

**`readOnly = true`** — 읽기 전용 트랜잭션이다. 스냅샷 비교를 안 해도 되니 메모리가 절약되고, 락을 안 걸거나 복제 DB로 라우팅해서 조회 성능이 좋아진다. 조회 메서드에만 쓴다.

**`rollbackFor = Exception.class`** — 기본적으로 `@Transactional`은 `RuntimeException` 발생 시에만 롤백한다. `IOException`, `SQLException` 같은 checked exception에서는 커밋해버린다. 모든 예외에 롤백하려면 이 옵션을 준다.

**`propagation`** — 이미 트랜잭션이 실행 중인 상태에서 또 다른 트랜잭션 메서드를 호출할 때 어떻게 처리할지 결정한다.
- `REQUIRED`: 기존 트랜잭션에 합류 (대부분 이걸 쓴다)
- `REQUIRES_NEW`: 기존 트랜잭션을 멈추고 독립 트랜잭션 생성 (예: 결제 실패 시 실패 내역은 따로 저장)

### 실무 적용 패턴

클래스에 읽기 전용 + 전체 롤백을 걸고, CUD 메서드에만 `@Transactional`을 따로 단다.

```java
@Service
@Transactional(readOnly = true, rollbackFor = Exception.class)
public class OrderService {

    public Object findOrderHistory(Long orderId) {
        // readOnly = true 적용 중
    }

    @Transactional
    public void createOrder(Long productId, int quantity) {
        // 쓰기 가능 트랜잭션 적용
    }
}
```

## 핵심 정리

- MyBatis는 자바 코드와 SQL을 분리하는 SQL Mapping Framework다.
- `#{ }`는 PreparedStatement 방식으로 SQL Injection에 안전하고, `${ }`는 문자열 치환 방식으로 칼럼명/정렬에만 쓴다.
- `<select>`는 `resultType`을, `<insert>/<update>/<delete>`는 `resultType`을 쓰지 않는다.
- `<resultMap>`은 칼럼명-필드명 불일치나 1:N 관계 매핑에 쓴다.
- 동적 쿼리: `<if>`로 조건부 조립, `<where>`/`<set>`으로 문법 에러 자동 수정, `<foreach>`로 컬렉션 순회
- 트랜잭션은 All or Nothing 원칙으로 여러 SQL을 하나로 묶는다.
- `@Transactional`은 선언적 트랜잭션 방식으로 실무에서 주로 쓴다.
- 클래스에 `readOnly = true, rollbackFor = Exception.class`를 걸고, CUD 메서드에만 `@Transactional`을 오버라이드하는 게 실무 패턴이다.
