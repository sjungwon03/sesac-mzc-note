# IoC와 DI

## 1. IoC와 Spring Container

### 1.1 등장 배경

**순수 자바 방식:**
```java
ProductService service = new ProductService();
```

**문제점:**

| 문제 | 설명 |
|------|------|
| 높은 결합도 | 클래스 간 의존도가 높음 |
| 연쇄 수정 | 한 클래스 수정 시 사용하는 다른 클래스도 수정 |
| 확장 불가 | 클래스가 수십 개일 경우 아키텍처적 재앙 |

**해결:**
- 객체 생성과 생명주기 관리 권한을 프레임워크에 위임

---

### 1.2 핵심 개념

**IoC (Inversion of Control, 제어의 역전):**
- 객체 제어권(생성, 관리, 소멸)이 개발자 → 프레임워크로 바뀐 현상
- 개발자가 `new`로 객체를 생성하지 않음

**스프링 컨테이너 (`ApplicationContext`):**
- IoC를 실현하는 객체 관리 컨테이너
- 컨테이너가 관리하는 객체를 **빈(Bean)**이라 부름

---

### 1.3 스프링 컨테이너 종류

| 컨테이너 | 설명 |
|----------|------|
| `BeanFactory` | 기본 컨테이너, 클라이언트 요청 시 빈 생성 |
| `ApplicationContext` | 트랜잭션, 메시지 처리 등 추가 기능, 빈 등록 즉시 로딩 |
| `GenericXmlApplicationContext` | XML 설정 파일(`<bean>`) 기반 빈 생성 |
| `AnnotationConfigApplicationContext` | 어노테이션(`@Configuration`, `@Bean`) 기반 빈 생성 |

---

### 1.4 특징

| 특징 | 설명 |
|------|------|
| 싱글톤 관리 | 기본적으로 빈을 단일 인스턴스로 재사용, 메모리 효율 |
| 결합도 완화 | 객체가 직접 다른 객체를 생성하지 않아 부품 교체처럼 유연 |

---

## 2. 빈 생성 방식

### 2.1 컴포넌트 스캔 (자동 등록)

**등장 배경:**
- 초기 스프링: XML에 모든 클래스를 `<bean>`으로 등록 → 클래스 증가 시 파악 어려움
- 스프링 2.5부터 컴포넌트 스캔 도입 → 어노테이션만으로 자동 등록

**방식:**
- 클래스 위에 `@Component` 어노테이션 부착
- 스프링 구동 시 자동으로 인스턴스화하여 빈 등록

**스테레오타입 어노테이션:**

| 어노테이션 | 용도 | 레이어 |
|-----------|------|--------|
| `@Component` | 일반 컴포넌트 | - |
| `@Controller` | 웹 요청 처리 | Presentation Layer |
| `@RestController` | REST API 응답 | Presentation Layer |
| `@Service` | 비즈니스 로직 | Service Layer |
| `@Repository` | 데이터베이스 접근 | Data Layer |

> 참고: 모든 스테레오타입 어노테이션은 내부에 `@Component` 포함

---

### 2.2 `@Configuration` + `@Bean` (수동 등록)

**필요성:**
- 라이브러리 객체 등 직접 수정 불가한 클래스는 `@Component` 부착 불가
- 수동 등록 필요

**방식:**
- 클래스에 `@Configuration` 부착
- 메서드에 `@Bean` 부착 → 메서드 반환 객체를 빈으로 등록

**특징:**
- 빈 등록 과정을 자바 코드로 직접 제어
- 조건부 등록, 초기화 메서드 호출 등 세밀한 제어 가능

**예시:**
```java
@Configuration
public class AppConfig {

  @Bean
  public DateTimeFormatter dateTimeFormatter() {
    return DateTimeFormatter.ofPattern("yyyy-MM-dd HH:mm:ss");
  }
}
```

| 항목 | 설명 |
|------|------|
| 메서드명 | 빈 이름 (`dateTimeFormatter`) |
| 반환타입 | 빈 타입 (`DateTimeFormatter`) |
| 반환객체 | 스프링 컨테이너에 저장할 빈 |

## 3. DI (Dependency Injection, 의존성 주입)

### 3.1 정의

- IoC 개념을 실제로 구현하는 구체적인 방법
- 스프링 컨테이너가 관리하는 객체 간 관계를 외부에서 주입
- 컨테이너에 저장된 빈을 자동으로 가져옴

---

### 3.2 주입 방식

**1. 필드 주입 (Field Injection):**
```java
public class ProductService {
  @Autowired
  private ProductRepository productRepository;
}
```

| 특징 | 설명 |
|------|------|
| 간결성 | 매우 높음 |
| 불변성 | 불가능 (final 불가) |
| 테스트 | 낮음 (Spring 없이 테스트 불가) |
| 권장 | ❌ 비권장 |

---

**2. 수정자 주입 (Setter Injection):**
```java
public class ProductService {
  private ProductRepository productRepository;

  @Autowired
  public void setProductRepository(ProductRepository productRepository) {
    this.productRepository = productRepository;
  }
}
```

| 특징 | 설명 |
|------|------|
| 간결성 | 보통 |
| 불변성 | 불가능 |
| 테스트 | 보통 |
| 사용 시기 | 런타임 시 객체 변경 필요 시 |
| 권장 | 🔺 선택적 사용 |

---

**3. 생성자 주입 (Constructor Injection):**
```java
public class ProductService {
  private final ProductRepository productRepository;

  // @Autowired는 스프링 4.3 이후 생성자 1개 시 생략 가능
  public ProductService(ProductRepository productRepository) {
    this.productRepository = productRepository;
  }
}
```

| 특징 | 설명 |
|------|------|
| 간결성 | 보통 |
| 불변성 | 가능 (final 사용) |
| 테스트 | 높음 (순수 자바 테스트 가능) |
| 권장 | ⭕️ 권장 (스프링 공식 권장) |

---

### 3.3 주입 방식 비교

| 방식 | 코드 간결성 | 불변성 (final) | 테스트 용이성 | 권장 여부 |
|------|-------------|----------------|---------------|-----------|
| 필드 주입 | 매우 높음 | 불가능 | 낮음 | ❌ 비권장 |
| 수정자 주입 | 보통 | 불가능 | 보통 | 🔺 선택적 |
| 생성자 주입 | 보통 | 가능 | 높음 | ⭕️ 권장 |

---

### 3.4 예시

```java
@Component
class Engine {
  public void start() {
    System.out.println("엔진 Start!");
  }
}

@Component
class Car {
  private final Engine engine;  // final로 불변성 확보

  @Autowired  // 생성자 1개이므로 생략 가능
  public Car(Engine engine) {
    this.engine = engine;
  }

  public void drive() {
    engine.start();
    System.out.println("자동차 주행 시작!");
  }
}
```

## 4. XML을 이용한 빈 생성 (스프링 레거시)

### 4.1 `root-context.xml`

**용도:**
- View와 무관한 객체 정의
- Service, Repository, DB 관련 빈 등록

**예시:**
```xml
<bean id="address" class="com.example.project.Address">
  <property name="zoneCode" value="12345" />
  <property name="jibunAddr" value="서울시 강남구 논현동" />
  <property name="roadAddr" value="서울시 강남구 강남대로" />
</bean>
```

> `<property>`는 Setter를 통해 값 주입

```java
@Setter
public class Address {
  private String zoneCode;
  private String jibunAddr;
  private String roadAddr;
}
```

---

### 4.2 `servlet-context.xml`

**용도:**
- 요청 관련 객체 정의
- Controller, ViewResolver, Interceptor 관련 빈 등록

**예시:**
```xml
<beans:bean class="org.springframework.web.servlet.view.InternalResourceViewResolver">
  <beans:property name="prefix" value="/WEB-INF/views/" />
  <beans:property name="suffix" value=".jsp" />
</beans:bean>
```

### 4.3 동작 순서

`servlet-context.xml` → `root-context.xml` 순으로 빈 탐색

## 핵심 요약

| 개념 | 핵심 내용 |
|------|-----------|
| IoC | 객체 제어권이 개발자 → 프레임워크로 역전 |
| Spring Container | 빈(Bean) 관리, 싱글톤 기본 |
| BeanFactory | 기본 컨테이너, 요청 시 빈 생성 |
| ApplicationContext | 추가 기능, 빈 등록 즉시 로딩 |
| 컴포넌트 스캔 | `@Component` 계열로 자동 등록 |
| @Configuration + @Bean | 수동 등록, 라이브러리 객체 등 |
| DI | 의존성 주입, 객체 간 관계 외부에서 주입 |
| 필드 주입 | 간단하지만 테스트 어려움, 비권장 |
| 수정자 주입 | 런타임 변경 가능, 선택적 사용 |
| 생성자 주입 | 불변성 확보, 테스트 용이, 권장 |
| XML 빈 등록 | 레거시 방식, root-context / servlet-context 분리 |
