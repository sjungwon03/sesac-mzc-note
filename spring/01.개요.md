# 자바 웹 개발 개요

## 1. 서블릿 (Servlet)

### 1.1 정의

- 클라이언트의 요청에 따라 동적으로 서비스를 제공하는 자바 클래스

### 1.2 특징

| 특징 | 설명 |
|------|------|
| 실행 위치 | 서버 쪽에서 실행 (Server Side) |
| WAS 필요 | 반드시 WAS 필요 (대표: Apache Tomcat) |
| 응답 생성 | 서블릿 내부에서 HTML 문서를 만들어 클라이언트 응답 |

### 1.3 동작 흐름

```
클라이언트 요청 → WAS → 서블릿 실행 → HTML 응답 → 클라이언트
```

## 2. Apache Tomcat

### 2.1 정의

- Apache Software Foundation에서 개발한 Java 기반 서블릿 컨테이너
- Java/Spring 웹 애플리케이션 구축에 필수적인 WAS
- 참고: Tomcat은 수컷 고양이를 의미

### 2.2 주요 용어

| 용어 | 설명 |
|------|------|
| `CATALINA_HOME` | 톰캣 설치 기본 경로 등록 시 사용하는 이름 |
| 컨텍스트 (Context) | 애플리케이션이 배포된 디렉터리 |
| `.war` 파일 | Web Application Archive, 웹 애플리케이션 압축 파일 |

### 2.3 배포 과정

| 단계 | 설명 |
|------|------|
| 1 | Java 웹 애플리케이션 완성 후 `.war` 파일로 빌드 |
| 2 | `.war` 파일을 `%CATALINA_HOME%\webapps`에 배포 |
| 3 | 배포 완료 시 `.war`와 같은 이름의 디렉터리(컨텍스트) 생성 |
| 4 | 접속 URL: `http://IP:8080/Context` |

### 2.4 배포 예시

- `galaxy.war` 파일을 로컬에 배포
- 접속 URL: `http://localhost:8080/galaxy`

## 3. `HttpServlet` 클래스

### 3.1 정의

- 웹 요청(HTTP Request)을 처리하고 동적 응답(HTTP Response)을 생성하기 위한 추상 클래스

### 3.2 요청/응답 구조

| 구분 | 방향 | 담당 클래스 |
|------|------|-------------|
| 요청 (Request) | 클라이언트 → 서버 | `HttpServletRequest` |
| 응답 (Response) | 서버 → 클라이언트 | `HttpServletResponse` |

### 3.3 특징

**1. 라이프사이클 관리**

| 항목 | 설명 |
|------|------|
| 관리 주체 | 서블릿 컨테이너(WAS, 예: Tomcat) |
| 생성 방식 | 개발자가 `new`로 생성하지 않음 |
| 라이프사이클 | `init()` → `service()` → `destroy()` |
| 객체 형태 | 싱글톤(Singleton)으로 관리 |

**2. 저수준(Low-Level) HTTP 제어**

- HTTP 요청 메시지를 파싱하여 자바 객체(`HttpServletRequest`)로 변환
- 자바 객체(`HttpServletResponse`)를 HTTP 응답 포맷으로 변환

### 3.4 주요 메서드

| 메서드 | 설명 |
|--------|------|
| `doGet(HttpServletRequest, HttpServletResponse)` | GET 요청 처리 |
| `doPost(HttpServletRequest, HttpServletResponse)` | POST 요청 처리 |
| `doDelete(HttpServletRequest, HttpServletResponse)` | DELETE 요청 처리 |

### 3.5 URL Mapping

**정의:**
- 서블릿을 브라우저에서 호출하기 위해 지정한 이름

**예시 1: 기본 매핑**
```java
public class TestServlet extends HttpServlet { }
```
- 호출 URL: `http://localhost:8080/galaxy/TestServlet`

**예시 2: 어노테이션 매핑**
```java
@WebServlet("/test")
public class TestServlet extends HttpServlet { }
```
- 호출 URL: `http://localhost:8080/galaxy/test`

**예시 3: 확장자 매핑 (실무 패턴)**
```java
@WebServlet("*.do")
public class UserServlet extends HttpServlet { }
```
- `*.do`로 끝나는 모든 요청을 `UserServlet`이 처리
- 예: `/login.do`, `/logout.do`, `/signup.do`

## 4. `HttpServletRequest` 클래스

### 4.1 정의

- 클라이언트 요청 정보를 처리하는 클래스
- WAS가 웹 브라우저로부터 요청을 받으면 객체 생성
- 요청 헤더, 파라미터, 쿠키 등 정보 처리

### 4.2 주요 메서드

| 메서드 | 반환 타입 | 설명 |
|--------|-----------|------|
| `getContextPath()` | `String` | 요청한 컨텍스트 URI 반환 |
| `getCookies()` | `Cookie[]` | 요청과 함께 보낸 쿠키 배열 반환 |
| `getHeader(String name)` | `String` | 특정 헤더 정보 문자열로 반환 |
| `getRequestURI()` | `String` | 요청 URL의 전체 경로 반환 |
| `getSession()` | `HttpSession` | 현재 요청과 연관된 세션 반환 |
| `getParameter(String name)` | `String` | 특정 파라미터 값 문자열로 반환 |
| `getParameterValues(String name)` | `String[]` | 특정 파라미터 값 문자열 배열로 반환 |

---

## 5. `HttpServletResponse` 클래스

### 5.1 정의

- 서버 응답 정보를 처리하는 클래스
- 출력 스트림 추출, 응답 주소 처리, 데이터 타입 및 문자셋 설정

### 5.2 주요 메서드

| 메서드 | 반환 타입 | 설명 |
|--------|-----------|------|
| `getWriter()` | `PrintWriter` | 클라이언트로 문자 전송 출력 스트림 반환 |
| `setContentType(String type)` | `void` | 응답 데이터 형식(MIME) 설정 |
| `addCookie(Cookie cookie)` | `void` | 응답에 쿠키 추가 (클라이언트에 저장) |
| `addHeader(String name, String value)` | `void` | 응답에 헤더 값 저장 |
| `sendRedirect(String location)` | `void` | 리다이렉트 URL 전송 |
| `setStatus(int sc)` | `void` | 응답 상태 코드 저장 |

## 6. Redirect vs Forward

### 6.1 Redirect

**정의:**
- 서버가 클라이언트에게 "새 주소로 이동하시오" 명령

**동작 방식:**

| 단계 | 주체 | 설명 |
|------|------|------|
| 1 | 클라이언트 | 주소 A로 요청 (`/create`) |
| 2 | 서버 | 상태 코드 `302` + `Location: /list` 응답 |
| 3 | 클라이언트 | `/list`로 재요청 |

**사용 시기:**
- Database 변경 작업(`INSERT`, `UPDATE`, `DELETE`) 후 반드시 사용

---

### 6.2 Forward

**정의:**
- WAS 내부에서 다른 서블릿으로 요청 제어권 넘김

**동작 방식:**

| 단계 | 주체 | 설명 |
|------|------|------|
| 1 | 클라이언트 | 주소 A로 요청 (`/list`) |
| 2 | 서버 A | 비즈니스 로직 처리 후 Forward |
| 3 | 서버 B | 이어 받은 요청 처리 후 응답 |

**사용 시기:**
- Database 변경 없는 작업(`SELECT`)이나 단순 페이지 이동

---

### 6.3 Redirect vs Forward 비교

| 비교 항목 | Redirect | Forward |
|-----------|----------|---------|
| 주체 | 클라이언트 재요청 | 서버 내부 위임 |
| 상태 코드 | `302` | 해당 없음 |
| URL 변경 | 변경됨 | 변경되지 않음 |
| 데이터 전달 (`setAttribute`) | 전달 안됨 | 전달됨 |
| 요청 횟수 | 2회 | 1회 |

### 6.4 코드 예시

```java
@RestController
public class TrafficController {

  // 리다이렉트: 클라이언트에게 302 응답
  @GetMapping("/test-redirect")
  public void doRedirect(HttpServletRequest request, HttpServletResponse response) throws IOException {
    request.setAttribute("message", "리다이렉트 전송 데이터");
    response.sendRedirect("/destination");
  }

  // 포워드: 서버 내부에서 제어권 넘김
  @GetMapping("/test-forward")
  public void doForward(HttpServletRequest request, HttpServletResponse response) throws ServletException, IOException {
    request.setAttribute("message", "포워드 전송 데이터");
    request.getRequestDispatcher("/destination").forward(request, response);
  }

  // 최종 목적지
  @GetMapping("/destination")
  public String destination(HttpServletRequest request) {
    String message = (String) request.getAttribute("message");
    return "목적지 도착 완료! (message: " + (message == null ? "null" : message) + ")";
  }
}
```

## 7. 쿠키(Cookie)와 세션(Session)

### 7.1 HTTP 프로토콜의 한계

| 특징 | 설명 |
|------|------|
| Connectionless (비연결성) | 요청/응답 후 연결 즉시 끊김 |
| Stateless (무상태성) | 연결 끊기면 서버가 클라이언트 상태 기억 못함 |

→ 상태 유지(Stateful)를 위해 쿠키와 세션 기술 등장

---

### 7.2 쿠키 (Cookie)

**정의:**
- 서버가 클라이언트 브라우저에 저장하는 작은 텍스트 파일 (4KB 제한)
- Key-Value 쌍 구성

**동작 흐름:**

| 단계 | 설명 |
|------|------|
| 1 | 클라이언트: 서버로 요청 |
| 2 | 서버: 응답에 `Set-Cookie: user=min` 포함 |
| 3 | 클라이언트: 쿠키 저장, 이후 요청 시 자동으로 쿠키 포함 |
| 4 | 서버: 쿠키 확인하여 사용자 식별 |

**특징:**

| 특징 | 설명 |
|------|------|
| 저장 위치 | 클라이언트 (웹 브라우저) |
| 보안성 | 낮음 (로컬에 노출, 탈취/위조 가능) |
| 속도 | 빠름 (서버 조회 불필요) |
| 만료 | 설정 기간 동안 유지 (브라우저 꺼져도 유지 가능) |
| 서버 부하 | 없음 |
| 담당 | `Cookie` 클래스 |
| 활용 | 오늘 하루 열지 않기, 광고 허용, 아이디 저장 등 |

---

### 7.3 세션 (Session)

**정의:**
- 민감한 상태 정보를 서버 측에서 안전하게 관리하는 기술

**동작 흐름:**

| 단계 | 설명 |
|------|------|
| 1 | 클라이언트: 서버로 요청 |
| 2 | 서버: 세션 저장소 생성 + Session ID 발급 → 쿠키로 전송 (`JSESSIONID=abc123`) |
| 3 | 클라이언트: 다음 요청부터 Session ID 쿠키 포함 |
| 4 | 서버: Session ID로 세션 저장소 조회 |

**특징:**

| 특징 | 설명 |
|------|------|
| 저장 위치 | 서버 (메모리/DB) |
| 보안성 | 높음 (서버 내부 관리, ID만 쿠키로 공유) |
| 속도 | 보통 (서버가 세션 저장소 조회 필요) |
| 만료 | 브라우저 종료 시 기본 만료 |
| 서버 부하 | 접속자 증가 시 서버 메모리 부담 |
| 담당 | `HttpSession` 인터페이스 |
| 활용 | 로그인 상태 유지 등 |

---

### 7.4 쿠키 vs 세션 비교

| 비교 항목 | 쿠키 (Cookie) | 세션 (Session) |
|-----------|---------------|----------------|
| 저장 위치 | 클라이언트 | 서버 |
| 보안성 | 낮음 | 높음 |
| 속도 | 빠름 | 보통 |
| 만료 | 설정 기간 유지 | 브라우저 종료 시 만료 |
| 서버 부하 | 없음 | 접속자 많을수록 부담 |
| 담당 | `Cookie` 클래스 | `HttpSession` 인터페이스 |

### 7.5 코드 예시

```java
@RestController
public class SessionCookieController {

  // 쿠키 생성
  @GetMapping("/api/v1/set-cookie")
  public String setCookie(HttpServletResponse response) {
    Cookie themeCookie = new Cookie("userTheme", "dark");
    themeCookie.setMaxAge(60 * 60);   // 유효 시간 1시간
    themeCookie.setPath("/");         // 모든 경로에서 쿠키 전송
    response.addCookie(themeCookie);
    return "쿠키 저장 성공!";
  }

  // 쿠키 읽기
  @GetMapping("/api/v1/get-cookie")
  public String getCookie(@CookieValue(name = "userTheme", required = false) String theme) {
    return theme == null ? "쿠키 없음" : "쿠키 값(userTheme): " + theme;
  }

  // 세션 로그인
  @GetMapping("/api/v1/login")
  public String login(HttpSession session) {
    session.setAttribute("loginUser", "홍길동");
    return "세션 로그인 성공";
  }

  // 세션 조회
  @GetMapping("/api/v1/dashboard")
  public String dashboard(@SessionAttribute(name = "loginUser", required = false) String user) {
    return user == null ? "로그인 정보 없음" : "현재 로그인한 유저: " + user;
  }

  // 세션 로그아웃
  @GetMapping("/api/v1/logout")
  public String logout(HttpSession session) {
    session.invalidate();  // 세션 완전 무효화
    return "로그아웃 완료";
  }
}
```

## 핵심 요약

| 개념 | 핵심 내용 |
|------|-----------|
| 서블릿 | 클라이언트 요청 처리 자바 클래스, WAS 필요 |
| Tomcat | Java 기반 서블릿 컨테이너, `.war` 배포 |
| HttpServlet | 웹 요청/응답 처리, 라이프사이클은 WAS가 관리 |
| HttpServletRequest | 클라이언트 요청 정보 처리 |
| HttpServletResponse | 서버 응답 정보 처리 |
| Redirect | 클라이언트 재요청, DB 변경 후 사용 |
| Forward | 서버 내부 위임, SELECT 등 읽기 작업 시 사용 |
| 쿠키 | 클라이언트 저장, 보안 낮음, 빠름 |
| 세션 | 서버 저장, 보안 높음, 메모리 부담 |
