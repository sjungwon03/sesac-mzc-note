# 9. Validation과 예외 처리

## 데이터 검증

전통적으로는 if문으로 검증했다. 필드가 많아지면 검증 코드가 비즈니스 로직보다 길어진다. 스프링 부트는 Bean Validation 어노테이션으로 이걸 자동화한다.

```java
// 전통 방식
if (dto.getEmail() == null) { ... }
if (dto.getEmail().contains("@")) { ... }

// Bean Validation
@Email
private String email;
```

## Spring Boot Validation 어노테이션

의존성 추가:

```groovy
implementation 'org.springframework.boot:spring-boot-starter-validation'
```

| 어노테이션 | 특징 | 비고 |
| ---------- | ---- | ---- |
| `@NotNull` | `null`만 차단 | `""`, `" "`은 통과 |
| `@NotEmpty` | `null`과 `""` 차단 | `" "`은 통과 |
| `@NotBlank` | `null`, `""`, `" "` 모두 차단 | |
| `@Max(value)` | 최댓값 지정 | |
| `@Min(value)` | 최솟값 지정 | |
| `@Email` | 이메일 형식 검증 | `@` 포함 여부 등 |
| `@Size(min, max)` | 길이/크기 제한 | |

`@NotBlank`가 가장 엄격하다. 실무에서는 `@NotBlank`를 주로 쓴다.

```java
public class JoinRequest {
    @NotBlank(message = "아이디는 필수 입력 값입니다.")
    @Size(min = 4, max = 12, message = "아이디는 4자 이상 12자 이하로 입력해 주세요.")
    private String username;

    @NotBlank(message = "비밀번호는 필수 입력 값입니다.")
    private String password;

    @NotBlank(message = "이메일은 필수 항목입니다.")
    @Email(message = "올바른 이메일 형식이 아닙니다.")
    private String email;

    @Min(value = 20, message = "성인(20세 이상)만 가입이 가능합니다.")
    private int age;
}
```

## `BindingResult`

검증 오류 정보를 담는 인터페이스다. `BindingResult`가 없으면 검증 실패 시 바로 예외가 튀어나와 400/500 에러가 뜬다. 선언해두면 에러 정보를 직접 다룰 수 있다.

**두 가지 제약:**
1. `@Valid`가 붙은 객체와 함께 써야 한다
2. 검증 대상 매개변수 **바로 뒤**에 위치해야 한다

```java
@PostMapping("/join")
public ResponseEntity<?> doJoin(
        @Valid @RequestBody JoinRequest joinRequest,
        BindingResult bindingResult  // 반드시 @Valid 바로 뒤
) {
    if (bindingResult.hasErrors()) {
        Map<String, String> errorDetailMap = new HashMap<>();
        for (FieldError error : bindingResult.getFieldErrors()) {
            errorDetailMap.put(error.getField(), error.getDefaultMessage());
        }
        return ResponseEntity.badRequest().body(errorDetailMap);
    }
    return ResponseEntity.ok(Map.of("status", "SUCCESS"));
}
```

## 글로벌 예외 처리

`BindingResult`를 매 컨트롤러마다 넣으면 보일러플레이트 코드가 반복된다. AOP 개념을 활용해 중앙에서 예외를 처리하면 깔끔해진다.

### `@RestControllerAdvice`

모든 컨트롤러에서 발생하는 예외를 감시하고 가로챈다. `@ControllerAdvice`는 뷰 이름을 반환하고, `@RestControllerAdvice`는 JSON을 반환한다.

### `@ExceptionHandler`

처리할 예외 타입을 지정한다.

```java
@RestControllerAdvice
public class GlobalExceptionHandler {

    // @Valid 검증 실패
    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<Map<String, String>> handleValidationExceptions(
            MethodArgumentNotValidException e
    ) {
        Map<String, String> errors = new HashMap<>();
        e.getBindingResult().getAllErrors().forEach(error -> {
            String fieldName = ((FieldError) error).getField();
            String errorMessage = error.getDefaultMessage();
            errors.put(fieldName, errorMessage);
        });
        return ResponseEntity.badRequest().body(errors);
    }

    // 나머지 모든 에러
    @ExceptionHandler(Exception.class)
    public ResponseEntity<Map<String, String>> handleAllExceptions(Exception e) {
        return ResponseEntity.internalServerError().body(Map.of(
                "errorCode", "SERVER_ERROR",
                "errorMessage", "서버 내부 오류"
        ));
    }
}
```

글로벌 예외 처리기를 만들면 컨트롤러에서 `BindingResult`를 생략할 수 있다.

```java
@PostMapping("/join")
public ResponseEntity<?> doJoinEnhanced(@Valid @RequestBody JoinRequest joinRequest) {
    return ResponseEntity.ok(Map.of("status", "SUCCESS"));
}
```

## 커스텀 예외 처리

`IllegalArgumentException`은 "뭔가 잘못됐다"는 것만 알 수 있다. `DuplicateUsernameException`은 "아이디가 중복됐구나"를 바로 알 수 있다. 이름 자체로 의도를 전달하는 게 커스텀 예외의 핵심이다.

### 1단계: 에러 코드 Enum

```java
public enum ErrorCode {
    DUPLICATE_USERNAME(HttpStatus.BAD_REQUEST, "AUTH-001", "이미 존재하는 아이디입니다."),
    USER_NOT_FOUND(HttpStatus.NOT_FOUND, "USER-002", "존재하지 않는 회원 정보입니다."),
    INVALID_PASSWORD(HttpStatus.UNAUTHORIZED, "AUTH-003", "비밀번호가 일치하지 않습니다.");

    private final HttpStatus httpStatus;
    private final String code;
    private final String message;

    ErrorCode(HttpStatus httpStatus, String code, String message) {
        this.httpStatus = httpStatus;
        this.code = code;
        this.message = message;
    }

    public HttpStatus getHttpStatus() { return httpStatus; }
    public String getCode() { return code; }
    public String getMessage() { return message; }
}
```

### 2단계: 커스텀 예외 클래스

부모 예외 클래스를 만들고, 실제 예외는 이걸 상속받는다.

```java
public class BusinessException extends RuntimeException {
    private final ErrorCode errorCode;

    public BusinessException(ErrorCode errorCode) {
        super(errorCode.getMessage());
        this.errorCode = errorCode;
    }

    public ErrorCode getErrorCode() { return errorCode; }
}

public class DuplicateUsernameException extends BusinessException {
    public DuplicateUsernameException() {
        super(ErrorCode.DUPLICATE_USERNAME);
    }
}
```

### 3단계: Service에서 예외 던지기

```java
@Service
public class UserService {
    public void registerUser(String username) {
        boolean isExist = "admin".equals(username);
        if (isExist) {
            throw new DuplicateUsernameException();
        }
    }
}
```

### 4단계: 글로벌 예외 처리기에서 처리

```java
@ExceptionHandler(BusinessException.class)
public ResponseEntity<Map<String, Object>> handleBusinessException(BusinessException e) {
    ErrorCode errorCode = e.getErrorCode();
    Map<String, Object> errorResponseBody = Map.of(
            "status", errorCode.getHttpStatus().value(),
            "errorCode", errorCode.getCode(),
            "errorMessage", errorCode.getMessage()
    );
    return ResponseEntity.status(errorCode.getHttpStatus()).body(errorResponseBody);
}
```

## 핵심 정리

- Bean Validation 어노테이션으로 검증 코드를 자동화한다.
- `@NotBlank`가 가장 엄격하고 실무에서 많이 쓴다.
- `BindingResult`는 `@Valid` 바로 뒤에 선언해야 검증 에러를 직접 다룰 수 있다.
- `@RestControllerAdvice` + `@ExceptionHandler`로 중앙 집중 예외 처리가 가능하다.
- 커스텀 예외는 이름으로 의도를 전달하고, ErrorCode Enum으로 에러 코드를 관리한다.
- 예외 처리 흐름: Service에서 throw → Controller 전달 → `@RestControllerAdvice`가 가로채서 응답
